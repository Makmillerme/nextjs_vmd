# Код-рев'ю: калькулятор (calculator-dialog + calculator-expression)

## Критичні / логічні проблеми

### 1. **Незбалансовані дужки: зайва `)` ігнорується**
- **Файл:** `calculator-expression.ts`, `evaluate()`.
- **Поведінка:** Вираз `1+2)` дає результат `3`. Закриваюча дужка без пари лише викликає `flushOps(true)` і не вважається помилкою.
- **Ризик:** Користувач може не помітити зайву `)` і отримати неочікуваний результат.
- **Рекомендація:** Відстежувати глибину дужок; при `closeParen` перевіряти `depth > 0`; якщо після обробки всіх токенів `depth !== 0` або була зайва `)`, повертати NaN або окремий код помилки.

### 2. **`getDisplayFromExpression` без неявного множення**
- **Файл:** `calculator-expression.ts`, рядок ~352.
- **Поведінка:** `getDisplayFromExpression(expr)` викликає лише `tokenize(expr)`, без `insertImplicitMultiplication`. Для рядка типу `25(6` токени — `[25, openParen, 6]`; "поточне число" — 6. Після неявного множення це не змінюється, але логіка відображення і обчислення розходяться (одна з функцій з неявним `*`, інша без).
- **Ризик:** Низький для поточного використання, але при подальших змінах можливі розбіжності.
- **Рекомендація:** Або завжди використовувати один пайплайн (tokenize → insertImplicitMultiplication) для відображення, або чітко задокументувати, що getDisplayFromExpression працює тільки з "сирим" виразом.

### 3. **Відсоток без бінарного оператора**
- **Файл:** `calculator-expression.ts`, `resolvePercent()`.
- **Поведінка:** `5%` дає `0.05`. Узгоджено з описом ("92% → 0,92").
- **Рекомендація:** Залишити як є; при розширенні вводу явно задокументувати семантику "просто B/100".

## Середні проблеми

### 4. **Наукова нотація у відповіді**
- **Файл:** `calculator-expression.ts`, `formatNumber()`.
- **Поведінка:** `formatNumber(1e20)` повертає `"1e+20"` (без заміни крапки на кому).
- **Рекомендація:** Для дуже великих/малих чисел форматувати через обмеження знаків або toExponential і показувати з комою.

### 5. **Подвійна кома в токенайзері**
- **Файл:** `calculator-expression.ts`, `tokenize()`.
- **Поведінка:** `3,,4` дає два числа без оператора → NaN. Подвійна кома не обробляється явно.
- **Рекомендація:** Або забороняти (вважати помилкою), або задокументувати.

### 6. **Мутація рефа під час рендеру**
- **Файл:** `calculator-dialog.tsx`, `handlersRef.current = { ... }`.
- **Поведінка:** Реф оновлюється на кожному рендері. Прийнятний патерн "останнє значення".
- **Рекомендація:** Залишити або перенести оновлення в useEffect для чистоти рендеру.

### 7. **`isCentered` і реф `hasDragged`**
- **Файл:** `calculator-dialog.tsx`.
- **Поведінка:** Оновлення `hasDragged.current` не викликає ре-рендер; позиція оновлюється в onDragMove. Після першого руху все коректно.
- **Рекомендація:** Додати коментар, що isCentered залежить від position для актуального UI.

### 8. **Немає повідомлення про помилку обчислення**
- **Файл:** `calculator-dialog.tsx`, `handleEquals()`.
- **Поведінка:** При NaN все скидається, показується "0".
- **Рекомендація:** Показувати "Помилка" або короткочасне повідомлення замість тихого скидання.

## Дрібні / стильові проблеми

### 9. **Дублювання скидання resultLine/resultValue**
- **Файл:** `calculator-dialog.tsx`.
- **Поведінка:** `setResultLine(""); setResultValue("");` повторюється в append, handleBackspace, handleClear, handleUnary, insertConstant, handleEquals.
- **Рекомендація:** Винести в хелпер `clearResult` і викликати скрізь.

### 10. **Іконки перестворюються кожен рендер**
- Компоненти Icon* визначені в модулі, викликаються без мемоїзації. Для малих SVG прийнятно.
- **Рекомендація:** Залишити або винести в окремий файл іконок.

### 11. **Кнопка закриття: focus:outline-hidden**
- Може приховувати outline. Переконатися, що focus ring видно для a11y.

### 12. **Порожній вираз при `=`**
- Можна не викликати evaluate при `!expression.trim()` і нічого не змінювати, щоб уникнути зайвих оновлень.

### 13. **Типізація handlersRef**
- `useRef<...>(null!)` — перевірка `if (!h) return` у keydown робить це безпечним. Можна тип з `| null` для ясності.
