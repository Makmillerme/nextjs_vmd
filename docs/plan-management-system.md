# План: управлінська система обліку авто

## Контекст

- **Стек:** Next.js 15, Better-Auth (admin plugin), Prisma/PostgreSQL, ролі: admin, manager, accountant, boss, user.
- **Поточний стан:** Сторінка «Управління» (`/management`) — один блок «Завантаження ВМД» з вимкненими кнопками. Облік авто на `/vehicles` повноцінний (таблиця, фільтри, sheet, медіа). Користувачі/ролі є в БД, адмін-плагін увімкнений, UI для управління користувачами відсутній.
- **Парсер:** Не входить у цей проект. Імпорт товарів — через API products або UI. Модель: Product.processedFileId для звʼязку з обробленим файлом.

---

## 1. Управління завантажувачем (парсером)

### Мета
Централізовано керувати пайплайном: джерело даних (пошта / ручне завантаження) → розпакування → парсинг ВМД → збереження в БД (Vehicle + прив’язка до обробленого файлу).

### Що має бути в системі (функціонал)

| Блок | Опис |
|------|------|
| **Запуск пайплайну** | Кнопки «7 днів», «30 днів», «Поточний місяць», «Свій діапазон» + підтвердження. Відправка запиту на бекенд (API або черга). |
| **Стан/черга** | Список запущених або запланованих завдань: дата запуску, діапазон дат, статус (pending / in_progress / done / failed), лог/помилка. |
| **Історія оброблених файлів** | Таблиця/список записів «оброблений файл» (якщо з’явиться модель ProcessedFile): оригінальна назва, дата обробки, кількість створених/оновлених авто, посилання на PDF. |
| **Налаштування** | (Опційно) Підключення пошти (IMAP/API), шляхи для PDF, ліміти частки запитів, увімкнення/вимкнення автоматичного запуску. |

### Модель даних (орієнтовно)

- У Prisma вже є `Vehicle.processedFileId` (Int?, nullable). Моделі `ProcessedFile` поки немає.
- **Рекомендація:** ввести модель `ProcessedFile` (id, originalName, storedPath?, processedAt, dateRangeFrom?, dateRangeTo?, status, errorMessage?, metadata JSON, createdAt) і зв’язок Vehicle → ProcessedFile. Це дасть історію завантажень і зрозумілий зв’язок «файл → авто».
- Якщо парсер буде зовнішнім сервісом (Python/Node), потрібен API у nextjs_vmd: наприклад `POST /api/management/ingest` (body: dateFrom, dateTo або file) → створює завдання/запис і повертає id; окремий endpoint для статусу/логу.

### Етапи реалізації

1. **Фаза 1 (мінімум для Управління)**  
   - Розблокувати UI на `/management`: кнопки періодів + «Свій діапазон» (DateRangePicker).  
   - По кліку викликати `POST /api/management/ingest` з параметрами періоду (поки що мок або заглушка з логом).  
   - Показувати тост «Завдання створено» / «Підключіть бекенд пайплайну».

2. **Фаза 2 (модель + історія)**  
   - Додати в Prisma модель `ProcessedFile`, міграція.  
   - API: створення запису ProcessedFile при «запуску», оновлення статусу/результату (вручну або з воркера).  
   - На сторінці Управління — секція «Історія завантажень»: таблиця (дата, період, статус, к-сть авто, дії).

3. **Фаза 3 (інтеграція з реальним парсером)**  
   - Підключити реальний пайплайн (пошта або файловий парсер): виклик з nextjs_vmd або черга (BullMQ/Inngest тощо).  
   - Оновлення статусу ProcessedFile з воркера; при успіху — створення/оновлення Vehicle з `processedFileId`.

## 2. Управління обліком авто (сторінка, поля, види)

### Мета
Керувати тим, як виглядає та як використовується сторінка обліку авто: які поля показуються, які колонки в таблиці, хто що може робити за ролями.

### Що має бути

| Блок | Опис |
|------|------|
| **Видимі колонки** | Збережені налаштування колонок таблиці (які колонки і в якому порядку) — вже частково є (наприклад вибір колонок у vehicles-page). Можна зберігати пресет по ролі або один глобальний пресет для користувача (localStorage або БД). |
| **Доступні поля** | Список полів Vehicle, які дозволені для перегляду/редагування. Можна обмежити по ролі (наприклад accountant бачить лише вартість і ВМД, manager — все крім частини фінансів). |
| **Фільтри за замовчуванням** | Збережені фільтри або «вид за замовчуванням» (наприклад «Мої на модерації») — опційно. |
| **Експорт** | Кнопка експорту таблиці (CSV/Excel) з видимими колонками — опційно. |

### Де це керується

- **Управління → підрозділ «Облік авто»:** форма/таблиця налаштувань: «Роль» → множина полів «видимі» / «редагувати», пресет колонок за замовчуванням.  
- Або окрема сторінка «Налаштування обліку» під Управлінням.  
- На `/vehicles` — читання цих налаштувань (з сесії/ролі) і обмеження колонок/полів у sheet.

### Етапи реалізації

1. **Фаза 1**  
   - Визначити список «полів обліку» з існуючої моделі Vehicle і типу VehicleDetailSheet.  
   - Збереження вибору колонок у localStorage (як зараз) залишити; опційно додати «Скинути до замовчувань для ролі».

2. **Фаза 2**  
   - Модель у БД: наприклад `AccountSetting` або використати існуючу (якщо є) — ключ `vehicles_columns`, значення JSON.  
   - Секція в Управлінні: «Колонки та поля за роллю» — вибір ролі, множина полів (видимість, можливість редагування).  
   - На сторінці vehicles: при відкритті sheet перевіряти права на поле (readonly для заборонених).

3. **Фаза 3**  
   - Експорт CSV з поточних даних таблиці з урахуванням видимих колонок і прав.

## 3. Користувачі та доступ

### Мета
Адміністратор може переглядати користувачів, змінювати ролі, банити/розбанити, запрошувати (якщо буде invite flow).

### Поточний стан

- Better-Auth + admin plugin: `adminRoles: ["admin"]`, `defaultRole: "user"`.  
- User: id, name, email, emailVerified, image, createdAt, updatedAt, role, banned, banReason, banExpires.  
- Ролі з `src/config/roles.ts`: admin, manager, accountant, boss, user.

### Що зробити

| Функція | Опис |
|--------|------|
| **Список користувачів** | Сторінка «Користувачі» (під Управлінням або окремий пункт меню для admin): таблиця (email, ім’я, роль, статус бану, дата реєстрації). |
| **Зміна ролі** | Вибір ролі з ROLES для обраного користувача, збереження через Better-Auth admin API або власний API з перевіркою session.role === 'admin'. |
| **Бан / розбан** | Встановлення banned, banReason, banExpires. Перевірка в auth: якщо user.banned — не дозволяти логін або показувати повідомлення. |
| **Запрошення** | (Опційно) Відправка magic link або запрошення на email з попередньо вибраною роллю. |

### API

- Better-Auth admin plugin вже дає endpoints для списку користувачів, оновлення ролі тощо — перевірити документацію і викликати з клієнта або через серверні route handlers.  
- Альтернатива: власні `GET /api/admin/users`, `PATCH /api/admin/users/[id]` (role, banned, banReason, banExpires) з перевіркою `session.user.role === 'admin'` і використанням prisma.

### Етапи реалізації

1. **Фаза 1**  
   - Сторінка `/management/users` (або таб на `/management`): тільки для ролі admin (перевірка в layout або page).  
   - Таблиця користувачів: дані з БД (prisma.user.findMany), колонки email, name, role, banned, createdAt.  
   - Дії: кнопка «Роль» → Select з ROLES → PATCH користувача (API або Better-Auth admin).

2. **Фаза 2**  
   - Модалка/сторінка редагування користувача: роль, бан (checkbox + причина + дата закінчення).  
   - У middleware або в Better-Auth: перевірка user.banned перед тим як вважати сесію валідною (або показувати сторінку «Обліковий запис заблоковано»).

3. **Фаза 3**  
   - Запрошення по email (magic link з defaultRole), історія запрошень — за потреби.

## Порядок впровадження (рекомендований)

1. **Користувачі та доступ** — база для подальших обмежень по ролях: список користувачів, зміна ролі, бан. Далі можна прив’язувати «Управління обліком» до ролей.  
2. **Управління завантажувачем** — розблокувати UI, заглушка API, потім модель ProcessedFile і історія; далі інтеграція з реальним парсером.  
3. **Управління обліком авто** — пресети колонок/полів по ролі і обмеження полів у sheet.

## Ризики та примітки

- **Парсер:** Не входить у проект. Product.processedFileId для звʼязку з обробленим файлом.
- **Права на маршрути:** Обмежити `/management` та `/management/users` тільки для admin; решту management — за бажанням (наприклад manager бачить тільки «Завантаження» та «Облік авто», без «Користувачі»).  
- **Serena:** Після реалізації кроків варто оновлювати контекст/документацію в Serena (наприклад опис нових API, сторінок, ролей).
