# План виправлення управління ролями

## Діагностика (результати аналізу)

### 1. API ролей (`/api/roles`, `/api/roles/[id]`)
- **GET/POST/PATCH/DELETE** перевіряють `session.user.role === "admin"` (requireAdmin).
- Якщо у сесії немає `user.role` або він не дорівнює `"admin"`, повертається **403** → список і деталі не завантажуються, таблиця порожня (крім системного рядка «Адмін»), Sheet при редагуванні порожній.

### 2. Джерело даних для таблиці
- Запит: `useQuery` → `fetchRoles()` → `GET /api/roles` → `data.roles ?? []`.
- При 401/403 `res.ok === false` → кидається помилка → у таблиці лише системна роль «Адмін».

### 3. Sheet і збереження
- При невдалому **GET /api/roles/[id]** (404/500/403) `roleDetail` лишався `undefined` → у Sheet передавався `null` → порожні поля без повідомлення.
- Після **POST** створення ролі права зберігалися окремим **PATCH**; результат PATCH не перевірявся → при помилці права не зберігалися, а користувач бачив «Роль створено».

### 4. Права доступу (permissions)
- Зараз перевіряється лише **роль адміна** (`user.role === "admin"`) для доступу до API ролей та UI управління.
- Права з таблиці **RolePermission** (БД) зберігаються і відображаються в UI, але **не використовуються** для обмеження доступу до інших API або сторінок (це майбутня доработка).

## Виправлення, які вже внесені

1. **createRole**  
   Після PATCH для збереження прав перевіряється `patchRes.ok`; при помилці кидається виключення з текстом від API → користувач бачить toast «Помилка збереження прав ролі» (або повідомлення від сервера).

2. **RoleDetailSheet**  
   - Додано проп **loadError** (текст помилки завантаження деталей).
   - Якщо при редагуванні деталі не завантажилися (`role === null` і є `loadError`), у Sheet показується повідомлення про помилку і кнопка «Зберегти» прихована.

3. **roles-management**  
   - З useQuery деталей ролі беруться **isError** та **error**.
   - Обчислюється **detailLoadError** і передається в **RoleDetailSheet** як **loadError**.
   - Користувач бачить причину, чому дані в Sheet порожні (наприклад, 403/404/500).

## Що перевірити вручну (чому «не працює»)

### A. Роль адміна в БД і в сесії
- У таблиці **user** (Prisma) має бути поле **role** (Better Auth admin plugin).
- Хоча б один користувач має мати **role = 'admin'** (наприклад, ваш обліковий запис).
- Перевірка в БД:
  ```sql
  SELECT id, email, role FROM "user" WHERE email = 'ваш@email.com';
  ```
- Якщо `role` NULL або не `admin`, оновіть:
  ```sql
  UPDATE "user" SET role = 'admin' WHERE email = 'ваш@email.com';
  ```
- Після цього вийдіть із системи та увійдіть знову, щоб сесія оновилася і `session.user.role` став `"admin"`.

### B. Міграції Better Auth
- Якщо поле `role` ще немає в таблиці `user`, виконайте міграції Better Auth:
  ```bash
  npx @better-auth/cli migrate
  ```
  або додайте поле вручну в міграцію Prisma і застосуйте її.

### C. Після виправлень коду
- Список ролей: має завантажуватися з API і показуватися в таблиці (плюс системний рядок «Адмін»).
- Відкриття ролі для редагування: деталі завантажуються з GET `/api/roles/[id]`, у Sheet відображаються назва, код, опис і права; при помилці завантаження показується **loadError**.
- Створення ролі: після збереження створюється запис і права зберігаються через PATCH; при помилці PATCH показується toast з помилкою.
- Редагування ролі: зміни зберігаються через PATCH, після успіху — invalidateQueries, дані оновлюються при наступному відкритті.

## Майбутні кроки (якщо потрібна перевірка прав з БД)

- Додати хелпер, який за `user.role` (код ролі) завантажує права з **RolePermission** для цієї ролі.
- Використовувати цей хелпер у middleware або в обгортках API для перевірки доступу за section/action (наприклад, `vehicles.view`, `users.edit`).
- У UI приховувати або блокувати кнопки/вкладки за тими ж правами.

Документація Better Auth Admin: https://www.better-auth.com/docs/plugins/admin
